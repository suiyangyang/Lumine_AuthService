# 第三方服务接入 `Lumine.AuthServer` 说明

## 1. 适用场景

`Lumine.AuthServer` 当前可以为第三方服务提供三类能力：

- **统一登录认证**：第三方服务把用户名、密码提交给认证服务，换取 JWT 访问令牌。
- **OIDC 身份信息**：支持 `openid profile email roles permissions` 等 scope，并可返回 `id_token` 与 `userinfo`。
- **统一令牌校验**：第三方服务使用相同的 `Jwt:Key` 和 `Jwt:Issuer` 验证该 JWT，从而识别调用人身份。

> 当前版本**已实现**登录发 `access_token` / `id_token`、`userinfo` 查询；**未提供**刷新 token、标准授权码模式、远程 introspection 等开放接口。

---

## 2. 当前服务已开放的认证接口

根据现有代码，认证服务当前开放的核心接口如下：

### 2.1 登录接口

- **URL**：`POST /api/Auth/login`
- **Controller**：`Lumine.AuthServer/Controllers/AuthController.cs`
- **请求体**：

```json
{
  "userName": "admin",
  "password": "123456",
  "scope": "openid profile email roles permissions",
  "clientId": "lumine-demo-client",
  "nonce": "sample-nonce-001"
}
```

- **成功响应**：

```json
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 28800,
  "scope": "openid profile email roles permissions",
  "id_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "user": {
    "id": "...",
    "userName": "admin",
    "email": "admin@lumine.local",
    "roles": ["Admin"],
    "permissions": ["users.view", "users.manage"]
  }
}
```

- **失败响应**：`401 Unauthorized`

### 2.2 UserInfo 接口

- **URL**：`GET /connect/userinfo`
- **鉴权**：`Authorization: Bearer {access_token}`
- **说明**：调用方必须在登录时申请 `openid` scope。
- **成功响应示例**：

```json
{
  "sub": "4f9e3a16-0d61-4de7-9b7f-6db8a2389f9a",
  "preferred_username": "admin",
  "name": "系统管理员",
  "nickname": "系统管理员",
  "email": "admin@lumine.local",
  "email_verified": false,
  "roles": ["Admin"],
  "permissions": ["users.view", "users.manage"]
}
```

### 2.3 OIDC Discovery 文档

- **URL**：`GET /.well-known/openid-configuration`
- **作用**：返回当前认证服务的 `issuer`、`jwks_uri`、`userinfo_endpoint`、支持的 `scope` 与 `claims`。

### 2.4 JWKS 公钥文档

- **URL**：`GET /jwks.json`
- **作用**：返回当前服务用于签发 `access_token` / `id_token` 的 RSA 公钥，第三方可据此做 JWT 验签。

## 3. JWT 令牌规则

认证服务在 `Program.cs` 中配置了 JWT 校验参数，在 `AuthController.cs` 中生成 token。

### 3.1 默认配置

来自 `Lumine.AuthServer/appsettings.json`：

```json
"Oidc": {
  "Issuer": "http://localhost:5115",
  "KeyId": "lumine-auth-rsa-key",
  "PrivateKeyPem": null,
  "GenerateDevelopmentKey": true
}
```

说明：

- `Issuer`：OIDC 对外发布的发行者地址，应与 discovery 文档中的 `issuer` 一致。
- `KeyId`：当前签名密钥标识，会出现在 JWT Header 的 `kid` 中。
- `PrivateKeyPem`：PEM 格式的 RSA 私钥，用于稳定签发 `access_token` / `id_token`。
- `GenerateDevelopmentKey`：未配置 `PrivateKeyPem` 时，是否自动生成临时开发密钥。

### 3.2 `Oidc:PrivateKeyPem` 配置示例

建议在生产环境提供固定 RSA 私钥；示例格式如下：

```json
"Oidc": {
  "Issuer": "https://auth.lumine.com",
  "KeyId": "lumine-auth-rsa-key-202603",
  "PrivateKeyPem": "-----BEGIN PRIVATE KEY-----\nMIIEvQIBADANBgkqhkiG9w0BAQEFAASCBKcwggSjAgEAAoIBAQD...\n...base64-content...\n-----END PRIVATE KEY-----",
  "GenerateDevelopmentKey": false
}
```

注意：

- JSON 中必须使用 `\n` 表示换行，不能直接粘贴多行文本。
- `PrivateKeyPem` 必须是 **PKCS#8 PEM** 格式。
- 若你使用环境变量，也可以写成 `Oidc__PrivateKeyPem`。

### 3.3 如何生成 PEM 私钥

可使用 `OpenSSL` 生成一对 `RSA 2048` 密钥：

```powershell
openssl genpkey -algorithm RSA -out lumine-auth-private.pem -pkeyopt rsa_keygen_bits:2048
openssl rsa -pubout -in lumine-auth-private.pem -out lumine-auth-public.pem
```

如果要放入 `appsettings.json`，需要把私钥文件转成单行并把换行替换为 `\n`。

### 3.4 生产建议

- 生产环境将 `GenerateDevelopmentKey` 设为 `false`。
- 生产环境必须提供固定 `PrivateKeyPem`，否则服务重启后旧 token 会失效。
- 如需轮换密钥，建议同时更新 `KeyId`，让第三方通过 `jwks.json` 识别新公钥。

### 3.5 当前 token 的 Claims

当前 `access_token` 会包含：

- `sub`：用户 ID
- `username`：用户名
- `role`：角色
- `permission`：权限
- `scope`：本次授权申请到的 scope

当请求中带 `openid` scope 时，还会返回 `id_token`。

### 3.6 scope 与 claims 对应规则

| scope | `id_token` / `userinfo` 返回内容 |
| --- | --- |
| `openid` | `sub`，并允许调用 `userinfo` |
| `profile` | `preferred_username`、`name`、`nickname` |
| `email` | `email`、`email_verified` |
| `roles` | `roles` |
| `permissions` | `permissions` |

### 3.7 当前 token 的有效期

- **8 小时**

### 3.8 当前校验规则

认证服务内部启用了如下校验规则：

- 校验 `Issuer`
- **不校验 `Audience`**
- 校验签名
- 校验过期时间

因此第三方服务接入时，至少要保证：

- 使用同一份 `issuer`
- 使用 `jwks.json` 中发布的 RSA 公钥完成验签

## 4. 第三方服务标准接入流程

推荐接入流程如下：

1. 用户先调用认证服务 `POST /api/Auth/login`
2. 认证服务返回 `access_token`，当申请了 `openid` 时同时返回 `id_token`
3. 第三方服务如需标准身份信息，可再调用 `/connect/userinfo`
4. 后续接口请求时，客户端在请求头中携带：

```http
Authorization: Bearer {token}
```

1. 第三方服务本地验证 JWT
2. 验证成功后，从 token 中读取用户 ID、用户名、角色、权限和 scope，作为当前登录人上下文

## 5. 第三方服务调用认证服务示例

假设认证服务地址为：

```text
http://localhost:5115
```

### 5.1 使用 cURL 登录

```bash
curl -X POST "http://localhost:5115/api/Auth/login" \
  -H "Content-Type: application/json" \
  -d "{\"userName\":\"admin\",\"password\":\"123456\"}"
```

### 5.2 使用 JavaScript / TypeScript 登录

```ts
type LoginResponse = {
  token: string;
  access_token: string;
  id_token?: string | null;
  scope?: string | null;
};

async function login() {
  const response = await fetch("http://localhost:5115/api/Auth/login", {
    method: "POST",
    headers: {
      "Content-Type": "application/json"
    },
    body: JSON.stringify({
      userName: "admin",
      password: "123456",
      scope: "openid profile email roles permissions",
      clientId: "lumine-demo-client",
      nonce: crypto.randomUUID()
    })
  });

  if (!response.ok) {
    throw new Error(`登录失败: ${response.status}`);
  }

  const data = (await response.json()) as LoginResponse;
  return data.access_token;
}
```

### 5.3 使用 C# `HttpClient` 登录

```csharp
using System.Net.Http.Json;

var httpClient = new HttpClient
{
    BaseAddress = new Uri("http://localhost:5115")
};

var response = await httpClient.PostAsJsonAsync("/api/Auth/login", new
{
    userName = "admin",
  password = "123456",
  scope = "openid profile email roles permissions",
  clientId = "lumine-demo-client",
  nonce = Guid.NewGuid().ToString("N")
});

response.EnsureSuccessStatusCode();

var result = await response.Content.ReadFromJsonAsync<LoginResult>();
var token = result!.Token;

public sealed class LoginResult
{
    public string Token { get; set; } = string.Empty;

  public string Access_Token { get; set; } = string.Empty;

  public string? Id_Token { get; set; }
}
```

## 6. 第三方 ASP.NET Core 服务如何校验该 Token

如果第三方服务本身也是 ASP.NET Core，推荐直接基于 OIDC Discovery / JWKS 接入 JWT Bearer。

### 6.1 配置示例

```csharp
using Microsoft.AspNetCore.Authentication.JwtBearer;
using Microsoft.IdentityModel.Tokens;

var builder = WebApplication.CreateBuilder(args);

var issuer = builder.Configuration["Oidc:Issuer"];

builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
    options.MetadataAddress = $"{issuer!.TrimEnd('/')}/.well-known/openid-configuration";
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = true,
            ValidateAudience = false,
            ValidateLifetime = true,
            ValidateIssuerSigningKey = true,
      ValidIssuer = issuer
        };
    });

builder.Services.AddAuthorization();

var app = builder.Build();
app.UseAuthentication();
app.UseAuthorization();

app.MapGet("/profile", (HttpContext httpContext) =>
{
    var user = httpContext.User;
    var userId = user.FindFirst("sub")?.Value;
    var userName = user.FindFirst("username")?.Value;

    return Results.Ok(new
    {
        userId,
        userName
    });
}).RequireAuthorization();

app.Run();
```

### 6.2 第三方服务配置文件示例

```json
{
  "Oidc": {
    "Issuer": "https://auth.lumine.com"
  }
}
```

> 这里的 `Issuer` 必须与 `Lumine.AuthServer` 的 `Oidc:Issuer` 保持一致，JWT 公钥会由 `jwks.json` 自动获取。

对于标准 OIDC 客户端，更推荐直接读取 `/.well-known/openid-configuration` 和 `jwks.json` 自动完成 issuer 与签名公钥配置，而不是手工共享密钥。

## 7. 第三方服务如何透传 Token 调用其他受保护接口

如果第三方服务需要把当前用户身份继续传给其他内部服务，可以透传原始 Bearer Token。

### 7.1 C# 示例

```csharp
var token = "这里放前端传入的JWT";

var request = new HttpRequestMessage(HttpMethod.Get, "http://other-service/api/orders");
request.Headers.Authorization = new System.Net.Http.Headers.AuthenticationHeaderValue("Bearer", token);

var response = await httpClient.SendAsync(request);
response.EnsureSuccessStatusCode();
```

### 7.2 Node.js 示例

```ts
const response = await fetch("http://other-service/api/orders", {
  headers: {
    Authorization: `Bearer ${token}`
  }
});
```

## 8. 关于“权限控制”的现状说明

当前项目中已经实现了基于权限名称的授权基础设施：

- `PermissionAttribute`
- `PermissionPolicyProvider`
- `PermissionHandler`

也就是说，在 `Lumine.AuthServer` 自己的接口中，可以通过下面这种方式声明权限：

```csharp
[Permission("Delete.User")]
public IActionResult DeleteUser()
{
    return Ok();
}
```

权限校验逻辑会：

1. 从 JWT 中读取 `sub`
2. 根据 `sub` 查用户
3. 查出用户绑定的角色
4. 从角色上聚合权限
5. 判断是否包含目标权限名

### 8.1 但第三方服务当前需要注意

当前版本**没有开放**以下接口：

- 获取当前用户权限列表接口
- 根据 token 远程校验权限接口
- 用户详情接口
- 角色/权限查询接口

因此，第三方服务如果要做**细粒度权限控制**，目前有两种方式：

- **方式 A：只使用登录能力**
  - 第三方服务只把认证服务当作发 token 的中心
  - 第三方服务自己维护权限模型

- **方式 B：统一权限中心化改造**
  - 由认证服务新增开放接口，例如：
    - `GET /api/Auth/me`
    - `GET /api/Auth/me/permissions`
    - `POST /api/Auth/check-permission`
  - 第三方服务通过这些接口进行远程权限判断

如果你的目标是“第三方系统完全复用这套权限体系”，建议优先补齐上述接口。

## 9. 推荐给第三方团队的最小接入规范

建议对外明确以下约定：

- **认证地址**：例如 `http://localhost:5115`
- **登录接口**：`POST /api/Auth/login`
- **认证头**：`Authorization: Bearer {token}`
- **JWT 签发方**：`lumine`
- **JWT 有效期**：8 小时
- **用户主键 Claim**：`sub`
- **用户名 Claim**：`username`
- **Audience**：当前不校验

## 10. 第三方服务接入建议

### 10.1 生产环境必须改进的点

从现有代码看，下面几点建议尽快处理后再开放给第三方：

- 当前登录逻辑是**明文比对密码**：

```csharp
if (user.PasswordHash != req.Password) return Unauthorized();
```

生产环境应改为：

- 使用密码哈希算法存储密码
- 登录时做哈希校验
- 不要把真实密码明文存储在数据库

### 10.2 建议补充的接口

为了让第三方服务更好接入，建议后续增加：

- `GET /api/Auth/me`：获取当前登录用户信息
- `GET /api/Auth/me/permissions`：获取当前用户权限列表
- `POST /api/Auth/refresh-token`：刷新 token
- `POST /api/Auth/check-permission`：远程检查某权限

---

## 11. 一句话总结

当前 `Lumine.AuthServer` 已经可以作为**统一登录和 JWT 签发中心**被第三方服务使用；如果要进一步作为**统一权限中心**，还需要补充查询当前用户与远程权限校验接口。
